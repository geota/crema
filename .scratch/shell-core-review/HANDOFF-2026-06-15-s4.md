## Non-obvious things found this session (read before touching these)

- **20 is a de1-app fix, not a wasm one.** The wasm bridge owns `core: CremaCore`
  directly, so its receiver is forced by the inner method's receiver; FFI is `&self`
  only because it wraps the core in `Mutex<CremaCore>`. The 6 `set_scale_*` were
  needlessly `&mut self` in **de1-app** (they borrow `&self.scale` + call the
  driver's `set_*_command(&self)`). `tare_scale` genuinely mutates
  (`Scale::tare(&mut self)` bumps the Decent sequence counter) → stays `&mut self`
  in wasm; can't match FFI's `&self` without a pointless `RefCell`.
- **22 avoids the cast entirely.** The old `(s*1000).round() … as u64` keeps a
  load-bearing `.round()` (compensates f32 error: `5.1 s → 5100 ms`; a naive
  `from_secs_f32(5.1)` truncates to 50 ds — a real regression). New code rounds in
  the float domain, then `Duration::from_secs_f64(f64::from(ms)/1000.0)` — the
  round-to-nearest-ns recovers exact ms, so **no float→int cast, no allow**.
  Verified parity across 14 inputs; only >1h values change (now clamp to 1h).
- **rustfmt sub-100 wrapping is real.** Beyond the s3 "only fix >100-col" rule:
  default rustfmt also wraps array/call literals over the **60-char** sub-heuristic
  (`fn_call_width`/`array_width`), even on sub-100 lines. I sidestepped by writing
  short test lines. The **pre-existing skew** (local rustfmt 1.9.0 vs CI) is only in
  **untouched** files — `de1-app/src/lib.rs:16,23`, `de1-ffi/src/lib.rs:1142–1208`,
  `de1-domain/{crema_profile,profile_roast,visualizer_*}`, `de1-scale/src/scale.rs`.
  Leave them; never `cargo fmt --all`.
